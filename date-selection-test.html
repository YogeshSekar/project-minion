Date Selection Flow Analysis

🔍 Potential Issues Identified

Issue 1: Double Event Handler Conflict

Problem: Both DateTimePicker and TaskSidePanel handle outside clicks, creating race conditions.

DateTimePicker: removed internal handler ✅
TaskSidePanel: still has outside click handler ⚠️

Issue 2: Portal Event Propagation

Problem: DateTimePicker uses createPortal to render outside component tree, but TaskSidePanel's outside click handler may not properly detect clicks inside the portal.

Solution: The portal element should be properly detected as "inside" the dropdown.

Issue 3: Immediate Closing After Selection

Problem: When user clicks a date, the following sequence occurs:

  1. User clicks date in DateTimePicker
  2. DateTimePicker calls onChange(date)
  3. TaskSidePanel calls setShowDueDatePicker(false)
  4. TaskSidePanel's outside click handler might also trigger
  5. Picker closes immediately, user sees no selection

Issue 4: Form Reset After Date Selection

Problem: The form initialization effect might be triggered after date selection due to dependency changes.

Current deps: [panelKey, isOpen] ✅ (should be stable)

But check: Is panelKey changing unexpectedly?

🚨 Most Likely Root Causes

Primary Suspect: Portal + Outside Click Conflict

The createPortal renders DateTimePicker outside TaskSidePanel's DOM tree, but the outside click handler uses document.querySelectorAll('.dropdown-container') which may not properly detect the portal content.

Secondary Suspect: Event Propagation Issues

The DateTimePicker has onClick={(e) => e.stopPropagation()} but this might interfere with proper event handling in the portal context.
